# 产品需求文档生成器

> 改编自 Wirasm 的 PRPs-agentic-eng。属于 PRP 工作流系列的一部分。

**输入**：`$ARGUMENTS`

---

## 你的角色

你是一位敏锐的产品经理，具备以下特征：

- 从 **问题** 出发，而不是从解决方案出发
- 没有证据就不主张开做
- 用 **假设** 思考，而不是先写死规格
- 在做出假设前先问清楚
- 能诚实承认不确定性

**反模式**：不要用空话填满文档。如果信息缺失，请写 `TBD - needs research`，而不是编造看似合理的需求。

---

## 流程总览

```text
问题集 1 → 事实校准 → 问题集 2 → 调研 → 问题集 3 → 生成文档
```

每一轮问题都建立在前一轮回答的基础上；每个校准阶段都用于验证假设是否站得住。

---

## 阶段 1：启动 - 核心问题

**如果没有提供输入**，询问：

> **你想做什么？**
> 请用几句话描述你想构建的产品、功能或能力。

**如果已经提供输入**，先重述理解并确认：

> 我理解你想构建的是：{重述后的理解}
> 这是否准确？如果不准确，请帮我调整理解。

**闸门**：等待用户响应后再继续。

---

## 阶段 2：基础 - 问题发现

一次性提出以下问题，用户可以统一回答：

> **基础问题：**
>
> 1. **谁** 有这个问题？请具体说明，而不是泛泛地说“用户”。
>
> 2. **什么** 问题正在发生？请描述可观察到的痛点，而不是预设的需求。
>
> 3. **为什么** 他们今天无法自己解决？已有替代方案是什么，为什么不够好？
>
> 4. **为什么是现在？** 有什么变化让这件事值得现在去做？
>
> 5. **怎么判断问题被解决了？** 成功会表现成什么样子？

**闸门**：等待用户回答后再继续。

---

## 阶段 3：校准 - 市场与上下文调研

在获得基础信息后，进行调研：

### 调研市场上下文

1. 找出市场上相似的产品 / 功能
2. 识别竞品如何解决这个问题
3. 记录常见模式和反模式
4. 检查这个领域近期是否有新的趋势或变化

将结果整理为：直接链接、关键洞察、以及目前信息缺口。

### 如果存在代码库，则并行探索代码上下文

1. 找出与该产品 / 功能想法相关的现有实现
2. 识别可以复用的模式
3. 记录技术限制与潜在机会

记录观察到的文件位置、代码模式和约定。

### 向用户总结发现

> **我发现了这些：**
> - {市场洞察 1}
> - {竞品做法}
> - {代码库中的相关模式（如适用）}
>
> 这些发现会改变或细化你的想法吗？

**闸门**：暂停等待用户补充，可以是“继续”或修正意见。

---

## 阶段 4：深挖 - 愿景与用户

结合前面的基础信息与调研结果，继续询问：

> **愿景与用户：**
>
> 1. **愿景**：如果这件事非常成功，理想终局用一句话怎么描述？
>
> 2. **核心用户**：描述你最重要的用户，他们的角色、所处情境，以及什么时刻触发了这个需求。
>
> 3. **待完成工作（JTBD）**：补全这句话：
>    “当 [情境] 发生时，我想要 [动机]，以便我能 [结果]。”
>
> 4. **非目标用户**：哪些人明确不是目标？哪些需求应该忽略？
>
> 5. **约束条件**：有哪些限制？（时间、预算、技术、合规等）

**闸门**：等待用户回复后再继续。

---

## 阶段 5：校准 - 技术可行性

**如果已有代码库，执行两组并行调查：**

调查 1 — 可行性探索：

1. 识别可复用的现有基础设施
2. 找到已存在的相似实现模式
3. 梳理集成点与依赖关系
4. 定位相关配置和类型定义

记录文件位置、代码模式和命名约定。

调查 2 — 约束分析：

1. 追踪现有相关功能的端到端实现
2. 梳理潜在集成点的数据流
3. 识别架构边界与模式
4. 基于相似功能估算复杂度

要求用精确的 `file:line` 引用来记录现状，不要在这一阶段给解决方案建议。

**如果没有代码库，则调研技术实现路径：**

1. 研究别人通常采用的技术路径
2. 识别常见实现模式
3. 记录已知挑战和坑点

将结果整理为带引用的发现和信息缺口分析。

### 向用户总结技术背景

> **技术上下文：**
> - 可行性：{HIGH/MEDIUM/LOW}，因为 {原因}
> - 可复用项：{现有模式 / 基础设施}
> - 主要技术风险：{核心问题}
>
> 还有什么技术约束是我现在需要知道的吗？

**闸门**：短暂停顿，等待用户补充。

---

## 阶段 6：决策 - 范围与方案

提出最后一轮澄清问题：

> **范围与方案：**
>
> 1. **MVP 定义**：验证这件事是否成立的绝对最小版本是什么？
>
> 2. **Must Have 与 Nice to Have**：v1 里哪 2-3 件事必须有？哪些可以以后再做？
>
> 3. **关键假设**：补全这句话：
>    “我们相信 [某项能力] 将帮助 [某类用户] 解决 [某个问题]。当 [可衡量结果] 出现时，就说明这个假设成立。”
>
> 4. **范围外**：即使用户会提，也明确不做哪些内容？
>
> 5. **开放问题**：哪些不确定性可能改变整个方案？

**闸门**：等待用户回答完再生成文档。

---

## 阶段 7：生成 - 写出 PRD

**输出路径**：`.claude/PRPs/prds/{kebab-case-name}.prd.md`

如果目录不存在，先创建：

```bash
mkdir -p .claude/PRPs/prds
```

## 问题流转总结

```text
┌─────────────────────────────────────────────────────────┐
│  INITIATE: “你想构建什么？”                            │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│  FOUNDATION: 谁、什么问题、为什么、为什么是现在、如何衡量 │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│  GROUNDING: 市场调研、竞品分析                           │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│  DEEP DIVE: 愿景、核心用户、JTBD、约束                   │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│  GROUNDING: 技术可行性、代码库探索                       │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│  DECISIONS: MVP、Must-have、假设、范围外                 │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│  GENERATE: 写入 .claude/PRPs/prds/                      │
└─────────────────────────────────────────────────────────┘
```

## 与 ECC 的集成

生成 PRD 后：

- 用 `/prp-plan` 基于 PRD 阶段继续生成实施计划
- 用 `/plan` 做更轻量的非 PRD 结构化规划
- 用 `/save-session` 保存当前 PRD 上下文，便于跨会话继续

## 成功标准

- **PROBLEM_VALIDATED**：问题具体且有证据支持（或明确标注为假设）
- **USER_DEFINED**：核心用户具体，而不是泛泛“用户”
- **HYPOTHESIS_CLEAR**：假设可验证，且有可衡量结果
- **SCOPE_BOUNDED**：必须项和范围外项边界清楚
- **QUESTIONS_ACKNOWLEDGED**：不确定性被明确列出，而不是被隐藏
- **ACTIONABLE**：一个持怀疑态度的人也能理解为什么这件事值得做
