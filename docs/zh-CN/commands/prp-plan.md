# PRP 规划

创建一份详细、可独立执行的实施计划，把单次完成某个功能所需的代码库模式、约定和上下文一次性收集完整。

**核心理念**：一份优秀的计划，应该让实现阶段无需再反复追问或补搜信息。所有模式、约定和坑点，都要在计划里一次讲清楚。

**黄金法则**：如果你在实现阶段还需要继续搜索代码库，那说明这些知识就应该现在先写进计划里。

---

## 阶段 0 — 检测

根据 `$ARGUMENTS` 判断输入类型：

| 输入模式 | 判断方式 | 动作 |
|---|---|---|
| 以 `.prd.md` 结尾的路径 | 指向 PRD 文件 | 解析 PRD，找到下一个待执行阶段 |
| 指向包含 “Implementation Phases” 的 `.md` 文件 | 类 PRD 文档 | 解析阶段并找出下一个待处理阶段 |
| 指向其他任意文件 | 参考文件 | 读取文件上下文，按自由描述处理 |
| 自由文本 | 功能描述 | 直接进入阶段 1 |
| 为空 / 空白 | 无输入 | 询问用户要规划什么功能 |

### PRD 解析（输入为 PRD 时）

1. 使用 `cat "$PRD_PATH"` 读取 PRD 文件
2. 解析 **Implementation Phases** 小节
3. 按状态查找阶段：
   - 查找 `pending` 阶段
   - 检查依赖链（某些阶段需要前置阶段已是 `complete`）
   - 选择 **下一个可执行的 pending 阶段**
4. 从选中的阶段提取：
   - 阶段名称与描述
   - 验收标准
   - 对前序阶段的依赖
   - 任何范围说明或约束
5. 把该阶段描述作为本次要规划的功能

如果没有剩余的 pending 阶段，则说明所有阶段均已完成。

## 阶段 1 — 解析

提取并澄清功能需求。

### 功能理解

从输入（PRD 阶段或自由描述）中识别：

- **做什么**：明确的交付物
- **为什么做**：用户价值或业务价值
- **谁来用**：目标用户 / 目标系统
- **放在哪里**：该功能位于代码库的哪一部分

### 用户故事

使用以下格式：

```text
作为一名 [用户类型]，
我希望 [获得某项能力]，
从而 [得到某项收益]。
```

### 复杂度评估

| 级别 | 指标 | 典型范围 |
|---|---|---|
| **Small** | 单文件、改动隔离、无新依赖 | 1-3 个文件，<100 行 |
| **Medium** | 多文件、沿用现有模式、引入少量新概念 | 3-10 个文件，100-500 行 |
| **Large** | 横切多个模块、新模式、外部集成 | 10+ 个文件，500+ 行 |
| **XL** | 架构级调整、新子系统、需要迁移 | 20+ 个文件，建议拆分 |

### 歧义闸门

如果以下任一项不清晰，**先暂停并询问用户**，不要继续：

- 核心交付物含糊不清
- 成功标准未定义
- 存在多种合理解释
- 技术路径有重大未知数

不要猜测。先问清楚。建立在错误假设上的计划，在实现阶段一定会出问题。

## 阶段 2 — 探索

直接搜索代码库，收集深度上下文。

### 代码库搜索（8 个维度）

针对以下每个维度，用 `grep`、`find` 和文件阅读来搜索：

1. **相似实现**：找出现有类似功能、接口、组件或模块。
2. **命名约定**：识别相关区域里的文件、函数、变量、类和导出命名方式。
3. **错误处理**：找出相似代码路径中错误如何捕获、传播、记录、返回给用户。
4. **日志模式**：识别记录什么、什么级别、使用什么格式。
5. **类型定义**：找出相关类型、接口、schema 及其组织方式。
6. **测试模式**：找相似功能如何测试，包括测试文件位置、命名、初始化/清理方式和断言风格。
7. **配置**：找相关配置文件、环境变量、功能开关。
8. **依赖**：识别相似功能使用的包、导入和内部模块。

### 代码库分析（5 条追踪）

阅读相关文件，梳理：

1. **入口点**：请求 / 动作如何进入系统并抵达要修改的区域？
2. **数据流**：数据如何沿相关路径流动？
3. **状态变化**：哪些状态会被修改、在哪里改？
4. **契约**：必须遵守哪些接口、API 或协议？
5. **模式**：使用了哪些架构模式（repository、service、controller 等）？

### 统一发现表

把发现整理成一个统一参考表：

| 类别 | 文件:行号 | 模式 | 关键片段 |
|---|---|---|---|
| 命名 | `src/services/userService.ts:1-5` | 服务用 camelCase，类型用 PascalCase | `export class UserService` |
| 错误 | `src/middleware/errorHandler.ts:10-25` | 自定义 AppError | `throw new AppError(...)` |
| ... | ... | ... | ... |

## 阶段 3 — 调研

如果该功能涉及外部库、外部 API 或不熟悉的技术：

1. 搜索官方文档
2. 查找用法示例与最佳实践
3. 识别版本相关的坑点

将每条发现整理成：

```text
KEY_INSIGHT: [学到的关键点]
APPLIES_TO: [影响计划的哪一部分]
GOTCHA: [警告事项或版本相关坑点]
```

如果功能只依赖成熟、已知的内部模式，可跳过此阶段并注明：
`No external research needed — feature uses established internal patterns.`

## 阶段 5 — 架构

### 战略设计

定义实施方案：

- **Approach**：高层策略，例如“沿用现有 repository 模式新增 service 层”
- **Alternatives Considered**：评估过哪些替代方案，以及为何不采用
- **Scope**：本次明确要做的边界
- **NOT Building**：明确不在本次范围内的内容，防止实现时范围膨胀

## 验证

在最终提交前，用以下清单验证计划质量：

### 上下文完整性
- [ ] 已发现并记录所有相关文件
- [ ] 已用示例记录命名约定
- [ ] 已记录错误处理模式
- [ ] 已识别测试模式
- [ ] 已列出依赖项

### 实现就绪度
- [ ] 每个任务都包含 ACTION、IMPLEMENT、MIRROR 和 VALIDATE
- [ ] 没有任何任务还需要额外搜索代码库
- [ ] 导入路径已经明确
- [ ] 必要时已记录 GOTCHA

### 模式忠实度
- [ ] 代码片段来自真实代码库，而不是臆造
- [ ] SOURCE 引用指向真实文件和行号
- [ ] 模式覆盖命名、错误、日志、数据访问和测试
- [ ] 新代码会与现有代码风格保持一致

### 验证覆盖
- [ ] 已指定静态分析命令
- [ ] 已指定测试命令
- [ ] 已包含构建验证

### UX 清晰度
- [ ] 已记录改动前后状态（或明确标记为 N/A）
- [ ] 已列出交互变化
- [ ] 已识别 UX 边界场景

### 无先验知识测试
一个不了解此代码库的开发者，应该只凭这份计划就能完成实现，而无需继续搜索代码库或追问问题。如果做不到，就补齐缺失上下文。
